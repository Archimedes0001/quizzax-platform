1.1 Project Charter

Project Overview

Project Title: Quizzax Assessment Platform

Description: Quizzax is a secure, web-based assessment platform designed for engineering students. It simplifies the process of taking continuous assessment (CA) quizzes by providing a responsive, interactive interface that offers instant feedback, detailed performance tracking, and a competitive leaderboard system. The platform replaces manual or paper-based testing with a modern, automated solution.

Objectives

  1. Automation: To automate the delivery and grading of engineering quizzes.
  2. Security: To ensure only authorized students (verified by matric number) can access assessments.
  3. Engagement: To increase student engagement through immediate feedback and gamified elements (leaderboards).
  4. Analytics: To provide students and instructors with actionable data on academic performance.

Scope

  • In-Scope: Student authentication, Quiz delivery (Timer, Randomization), Performance dashboard, Leaderboards, 4 core subjects.
  • Out-of-Scope: Instructor admin panel (Phase 1), Essay grading (AI), Payment processing.

Phase 2: System Design

2.1 System Architecture

Quizzax employs a 3-Tier Client-Server Architecture:

  1. Presentation Tier (Client): Vanilla JS SPA (Single Page Application).
  2. Application Tier (Server): Node.js/Express REST API.
  3. Data Tier (Database): MongoDB Atlas Cloud Database.

Tech Stack Justification

  • Frontend: Vanilla JS + HTML5 (Lightweight, < 2s load time).
  • Backend: Node.js (Non-blocking I/O for concurrency).
  • Database: MongoDB (Flexible schema for quiz documents).
  • Hosting: Render (Free tier, GitHub CI/CD).

2.2 Database Design

User Collection Schema

Field Type Description
_id ObjectId Unique Primary Key
matricNumber String Student ID (Unique)
name String Full Name
department String Academic Dept
history Array Embedded list of past attempts

Normalization Strategy

The design adheres to NoSQL best practices (approx. 2NF). We intentionally embed history and questions arrays rather than creating separate tables (as in 3NF SQL) to optimize read performance. This ensures we can fetch a student's full profile or a complete quiz in a single database query, meeting our response time requirement of < 200ms.

Phase 3: Implementation

3.2 Implementation Report

Key Algorithms

Fisher-Yates Shuffle

Used to randomize question order to prevent cheating.

function shuffle(array) {
  for (let i = array.length - 1; i > 0; i--) {
    const j = Math.floor(Math.random() * (i + 1));
    [array[i], array[j]] = [array[j], array[i]];
  }
}

Singleton Pattern (AppState)

Used on the client-side to maintain a single source of truth for the active user session and quiz timer state, ensuring consistency across the SPA views.

Phase 5: Deployment

5.1 Deployment Guide

Render.com Deployment (Cloud)

  1. Create Account: Sign up at dashboard.render.com.
  2. New Web Service: Click "New +" -> "Web Service".
  3. Connect Repo: Select "Build and deploy from a Git repository" -> `quiz_app`.
  4. Configuration:
    • Runtime: Node
    • Build Command: npm install
    • Start Command: npm start
  5. Environment Variables:
    • MONGO_URI: mongodb+srv://...
    • NODE_ENV: production

5.2 User Manual

  • Logging In: Enter your assigned Matric Number.
  • Taking a Quiz: Select a subject. You have 15 minutes to answer 50 questions.
  • Navigation: Use "Next" / "Prev" buttons. Use the Flag icon to mark questions for review.
  • Results: Your score is shown immediately. Check the "Performance" tab for your CA history.

FAQ

Q: Can I pause the timer?
A: No, it mimics exam conditions.

Q: What if I lose internet?
A: The system auto-saves your progress. Reconnect and refresh to resume.
