1.0 Project Charter

1.1 Executive Summary

The Quizzax Assessment Platform is a response to the growing need for automated educational tools in the engineering faculty. Traditional paper-based continuous assessments (CAs) are logistically burdensome, prone to grading errors, and provide delayed feedback to students. Quizzax solves these problems by offering a secure, real-time, web-based assessment environment.

1.2 Problem Statement

Currently, the department faces the following challenges:

  • Manual Grading: Grading 200+ scripts for 4 subjects takes weeks.
  • Paper Waste: Printing thousands of pages for tests is environmentally and financially costly.
  • Lack of Analytics: Students do not have access to their historical performance data to track improvement.

1.3 Proposed Solution

Quizzax provides:

  • Instantaneous Grading: Results are computed in milliseconds.
  • Dynamic Question Pooling: Each student receives a unique subset of questions, minimizing malpractice.
  • Longitudinal Tracking: A "Performance" dashboard that aggregates scores over the semester.

2.0 Requirements Specification (SRS)

2.1 Functional Requirements (Detailed)

The following requirements outline the specific behaviors of the system.

FR-01: Student Authentication

Logic Flow:
  • Input: Student enters 10-digit Matriculation Number (e.g., U24ICE1001).
  • Process: Server queries MongoDB `users` collection. detailed lookup.
  • Output (Success): Redirects to Dashboard, stores Session in LocalStorage.
  • Output (Failure): Displays "Unauthorized Access" toast error.

FR-05: Quiz Generation Algorithm

Logic Flow:
  • Input: Student selects "Engineering Mathematics".
  • Process: 1. Fetch all available questions for subject. 2. Apply Fisher-Yates Shuffle. 3. Slice the first 50 indices.
  • Output: A JSON array of 50 Question Objects sent to client.

FR-06: Timer Enforcement

Logic Flow:
  • Input: Quiz Load Event.
  • Process: `setInterval` runs every 1000ms, decrementing `remainingTime` variable.
  • Constraint: IF `remainingTime <= 0`, TRIGGER `submitQuiz()`.
  • Output: UI updates "14:59", "14:58"...

2.2 Non-Functional Requirements

NFR-01: Scalability

The system is designed to handle the entire 100-level cohort (approx 250 students) taking a quiz simultaneously. This is achieved via Node.js's asynchronous event loop.

NFR-02: Security

To prevent "Network Tab Cheating", the correct answers are NOT sent to the client during the quiz. Use of `httpOnly` cookies was considered but `localStorage` was chosen for simplicity in Phase 1.

const cors = require('cors');
require('dotenv').config();

const app = express();
app.use(cors());
app.use(express.json());

// Database Connection
mongoose.connect(process.env.MONGO_URI)
  .then(() => console.log('‚úÖ Connected to MongoDB Atlas'))
  .catch(err => console.error('‚ùå DB Connection Error:', err));

// Login Endpoint
app.post('/api/login', async (req, res) => {
    const { matricNumber } = req.body;
    const user = await User.findOne({ matricNumber });
    
    if (!user) {
        return res.status(404).json({ error: 'Student not found in registry' });
    }
    res.json({ 
        message: 'Login successful', 
        user: { 
            name: user.name, 
            dept: user.department 
        } 
    });
});

8.0 Detailed Deployment Guide

8.1 MongoDB Atlas Creation (Database Setup)

The database is hosted on MongoDB Atlas (Cloud). Here is how it was created step-by-step:

Step 1: Cluster Provisioning

1. Navigate to mongodb.com/atlas and sign in.

2. Click "Create" Button.

3. Select "M0 Sandbox" (Free Tier).

4. Provider: AWS; Region: N. Virginia (us-east-1).

5. Click "Create Cluster". (Wait 3-5 mins).

Step 2: specific Database Access

1. Go to "Database Access" tab.

2. Click "Add New Database User".

3. Username: admin_alpha, Password: securePassword123.

4. Role: "Atlas Admin".

Step 3: Network Access (Critical)

1. Go to "Network Access" tab.

2. Click "Add IP Address".

3. Select "Allow Access from Anywhere" (0.0.0.0/0). NOTE: This ensures our Render server can connect to it.

Step 4: Connection String

1. Click "Connect" on the Cluster dashboard.

2. Select "Drivers" (Node.js).

3. Copy string: mongodb+srv://admin_alpha:securePassword123@cluster0.mongodb.net/?retryWrites=true&w=majority

8.2 Render Deployment (Application Setup)

We used Render.com for hosting the Node.js API and Client.

Step 1: Service Creation

1. Log in to Render.com via GitHub.

2. Click "New" -> "Web Service".

3. Select the quizzax-platform repository from the list.

Step 2: Configuration

1. Name: quizzax-prod

2. Environment: Node

3. Build Command: npm install (Installs dependencies from package.json)

4. Start Command: node server/index.js (Starts the Express app)

Step 3: Environment Variables

This is where we secure our keys. Under "Environment":

  • Key: MONGO_URI | Value: (The string copied from Atlas)
  • Key: NODE_ENV | Value: production
