1.0 Project Charter

1.1 Executive Summary

The Quizzax Assessment Platform represents a significant technological leap for the Department of Engineering. In an era where digital transformation is paramount, the reliance on paper-based continuous assessments (CAs) has become a bottleneck for both instructors and students. Quizzax addresses this by providing a unified, secure, and automated web-based environment for conducting quizzes. This report details the entire Software Development Lifecycle (SDLC) from conception to deployment, demonstrating adherence to modern engineering practices.

1.2 Problem Statement & Motivation

The current manual assessment process is plagued by several critical inefficiencies:

  1. Turnaround Time: Grading 200+ scripts manually typically takes instructors 2-3 weeks, delaying feedback that is crucial for student learning correction.
  2. Resource Wastage: Printing question papers and answer booklets consumes significant paper and toner resources, increasing departmental operational costs.
  3. Data Fragmentation: Student grades are often stored in disparate Excel sheets or physical ledgers, making longitudinal performance tracking impossible for students.
  4. Malpractice Vulnerability: Identical paper questions in a crowded hall makes invigilation difficult and facilitates copying.

Motivation: The motivation behind Quizzax is to eliminate these pain points by automating the "Delivery -> Submission -> Grading -> Recording" pipeline. By doing so, we free up instructor time for teaching and give students agency over their academic progress through immediate data transparency.

1.3 Project Scope

In-Scope (Delivered)

  • Student Authentication: Secure login mechanism validating against a pre-authorized matriculation list.
  • Adaptive Quiz Engine: Dynamic generation of 50-question sets from a subject pool using randomization algorithms.
  • Time Management: Strict server-side and client-side synchronized countdown timers (15 minutes).
  • Auto-Submission: Fail-safe mechanisms to submit quizzes automatically when time expires.
  • Performance Dashboard: A visualization of student grades across different subjects (CA1, CA2, etc.).
  • Leaderboard: A gamified ranking system to encourage healthy competition.

Out-of-Scope (Future Phases)

  • Instructor Admin Panel: The ability for lecturers to upload questions via CSV (Phase 2).
  • Essay Grading: AI-based assessment of non-objective questions.
  • Payment Integration: Collecting departmental fees.

2.0 Requirements Specification (SRS)

2.1 Functional Requirements

This section outlines the specific inputs, processes, and outputs for the system's core functions. These requirements are prioritized using the MoSCoW method.

FR-01: Restricted Access Login (Must Have)

Detailed Flow

The system must ensure that only authorized students can access the platform.

  • Input: User enters matric number (e.g., 'U24ICE1024') on the login screen.
  • Process: 1. Client inputs are sanitized to prevent NoSQL injection. 2. A POST request is sent to `/api/login`. 3. Server queries the `users` collection for an exact match.
  • Output (Success): 200 OK Response. Client receives user object. Redirection to Dashboard.
  • Output (Fail): 403 Unauthorized. "Matric number not found" toast message displayed.

FR-02: Randomized Question Delivery (Must Have)

Detailed Flow

To prevent cheating, no two students should have the exact same question order.

  • Input: Student clicks a Subject Card (e.g., "Fluid Mechanics").
  • Process: 1. Server fetches the Subject Document containing 100+ questions. 2. The Fisher-Yates Shuffle Algorithm is applied to the questions array. 3. The system slices the first 50 indices `questions.slice(0, 50)`.
  • Output: A JSON array of 50 unique questions is delivered to the client state manager.

FR-03: Session Persistence (Must Have)

Detailed Flow

Students must not lose progress if the browser is accidentally closed or refreshed.

  • Mechanism: LocalStorage + Server Sync.
  • Process: 1. Every 30 seconds (or on answer change), the client saves `currentQuestionIndex`, `answers`, and `timer` to LocalStorage. 2. On page load (`init()`), the app checks for existing LocalStorage data.
  • Output: User is seamlessly returned to Question #XX with the correct time remaining.

FR-04: Automated Grading (Must Have)

Detailed Flow

Grades must be calculated instantly without human intervention.

  • Input: User submits answers array `['A', 'C', 'B', null, ...]`.
  • Process: 1. Client compares user answers against the cached valid answer key (Client-side implementation for Phase 1). 2. Score = `(Correct Answers / Total Questions) * 100`. 3. Computed score is sent to `/api/submit` for permanent storage.
  • Output: Result Page displays "Score: 84%".

2.2 Non-Functional Requirements

  • NFR-01 Performance: The system shall load the dashboard in under 2 seconds on a standard 4G network.
  • NFR-02 Scalability: The Node.js server shall handle at least 50 concurrent connections (single class size) without crashing.
  • NFR-03 Reliability: The system availability should be 99.9% during the scheduled exam window (hosted on Render Cloud).
  • NFR-04 Usability: The interface must follow Glassmorphism design principles to ensure a modern, appealing aesthetic that reduces test anxiety.

3.0 System Design

3.1 System Architecture

Quizzax utilizes a Three-Tier MERN Architecture (MongoDB, Express, React-concept/Vanilla JS, Node).

Tier 1: Presentation Layer (Client)

Built with Vanilla JavaScript (ES6+), HTML5, and CSS3. We avoided heavy frameworks like React for this iteration to ensure maximum performance and minimal bundle size. The client functions as a Single Page Application (SPA), dynamically updating the DOM without full page reloads.

Tier 2: Logic Layer (Server)

Built with Node.js and Express. It serves as a RESTful API. It handles:

  • Routing (endpoints for /login, /quiz, /submit).
  • Business Logic (Validation, Error Handling).
  • Database Interaction (Mongoose ORM).

Tier 3: Data Layer (Database)

Built on MongoDB Atlas (Cloud). A NoSQL document store was chosen over SQL because:

  • Schema Flexibility: Quiz questions can have varying options or metadata without altering table structures.
  • JSON Native: Data stored in BSON maps directly to the JSON objects used in our JavaScript frontend.

3.2 Database Schema Design

The `User` schema is the core of our data model. It embeds the `Quiz History` to optimize read performance (Data Locality).


// models/User.js Schema Definition
{
  matricNumber: String (Unique, Indexed),
  name: String,
  department: String,
  history: [
    {
      subject: String,
      score: Number,
      totalQuestions: Number,
      date: Date
    }
  ],
  activeSession: {
    // Stores temporary state for crash recovery
    subject: String,
    answers: Object,
    timer: Number
  }
}

Entity-Relationship Logic

Since we are using NoSQL, we denormalized the relationship between `Students` and `Results`. Instead of a separate `Results` table joined by foreign key, we store results inside the Student document. This means fetching a student profile automatically fetches their entire history in one query.

4.0 User Interface Design

4.1 Design Philosophy: Glassmorphism

We adopted the "Glassmorphism" trend to create a premium, modern feel. Key characteristics implemented include:

  • Translucency: Background-blur (backdrop-filter: blur(16px)) on cards to create a frosted glass effect.
  • Vivid Colors: Deep blue and purple gradients in the background to emphasize depth.
  • Hierarchy: Use of white borders with low opacity to define boundaries without heavy lines.

4.2 Wireframe Descriptions

4.2.1 Login Screen

A centered card layout. To solve the "Scrollbar Flicker" issue (Bug #142), we locked the viewport height (`100vh`) and hid overflow. The input field uses floating labels for a clean look.

4.2.2 Quiz Interface

Designed for focus. We utilized a Two-Column Layout on Desktop:

  • Left Sidebar: Contains the Question Grid (1-50) allowing random navigation. Questions are color-coded (Green=Answered, Yellow=Flagged, Gray=Unseen).
  • Main Area: Displays the Question Text and large, touch-friendly Option Buttons.

Mobile Responsiveness: On strict mobile viewports (< 768px), the Sidebar collapses into a drawer or moves to the bottom to prioritize the question text.

This file bootstraps the Express application and defines the API routes.


const express = require('express');
const mongoose = require('mongoose');
const cors = require('cors');
require('dotenv').config();

const app = express();
app.use(cors());
app.use(express.json());

// CONNECT TO DATABASE
mongoose.connect(process.env.MONGO_URI)
    .then(() => console.log('MongoDB Connected'))
    .catch(err => console.log(err));

// LOGIN ROUTE
app.post('/api/login', async (req, res) => {
    const { matricNumber } = req.body;
    // Query DB
    const user = await User.findOne({ matricNumber });
    if (!user) return res.status(403).json({ error: "Unauthorized" });
    
    res.json({ user });
});

// START SERVER
const PORT = process.env.PORT || 5000;
app.listen(PORT, () => console.log(`Server running on port ${PORT}`));

5.3 The Fisher-Yates Shuffle

The core of our randomization engine. This algorithm acts in O(n) time complexity, swapping elements from the back of the array to the front.


function shuffle(array) {
    for (let i = array.length - 1; i > 0; i--) {
        const j = Math.floor(Math.random() * (i + 1));
        // Array Destructuring Swap
        [array[i], array[j]] = [array[j], array[i]];
    }
    return array;
}

8.0 Deployment Guide

This section details the exact steps taken to deploy the application to the live internet.

8.1 Part A: MongoDB Atlas Creation

Goal: Set up a cloud database to store user records.

Step 1: Cluster Provisioning

We navigated to MongoDB Atlas and created a new account. We selected the Shared (Free) Tier to avoid costs during development. We chose AWS (us-east-1) as the region for low latency.

Step 2: Database User

We created a database user `admin_main` with the password `securePassword123` (placeholder). This user has `readWriteAnyDatabase` privileges.

Step 3: Network Access (IP Whitelist)

We navigated to "Network Access" and clicked "Add IP Address". We selected "Allow Access from Anywhere" (0.0.0.0/0). Ideally, we would whitelist only the Render server IP, but Render IPs are dynamic, so this global allowance is necessary for the free tier.

Step 4: Connection String

We obtained the SRV connection string: mongodb+srv://admin_main:password@cluster0.mongodb.net/?retryWrites=true&w=majority.

8.2 Part B: Render.com Hosting

Goal: Host the Node.js API and Client artifacts.

Step 1: Connect GitHub

On the Render Dashboard, we clicked "New Web Service" and authorized Render to access our GitHub account. We selected the `quizzax-platform` repository.

Step 2: Build & Start Commands

We configured the environment:

  • Build Command: npm install (Installs dependencies).
  • Start Command: npm start (Runs node server/index.js).
Step 3: Environment Variables

This is the most critical step for security. We manually added the variables:

  • MONGO_URI = (The Connection String from Part A)
  • NODE_ENV = production
Step 4: Verify Deployment

After clicking "Deploy", we monitored the logs. The message "Connected to MongoDB" confirmed success. The app is now live at https://quizzax.onrender.com.

9.0 User Manual & Maintenance

9.1 User Manual

Logging In

Open the URL. Enter your Matric Number. If you are authorized, you will enter the dashboard. If not, contact the administrator.

Taking a Quiz

Click "Start Quiz". The timer will begin immediately. You cannot pause it. Click options to select answers. Use "Next/Prev" to navigate. Click "Submit" when done.

9.2 Maintenance Procedures

Database Backup

MongoDB Atlas performs automated daily snapshots. In case of data corruption, Navigate to Atlas > Collections > Backups > Restore.

Adding New Students

Currently, this is done by updating the `students` array in `server/utils/studentData.js` and pushing the changes to Git. The server will auto-sync on restart.
